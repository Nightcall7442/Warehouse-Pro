# Миграции базы данных — как теперь работать

С этого момента изменения схемы (`db/schema.ts`) проходят через **миграции**
(`drizzle-kit generate` + `drizzle-kit migrate`), а не через `db push`.

## Почему это поменялось

`db push` напрямую сравнивает `schema.ts` с реальной БД и применяет разницу —
без истории, без возможности откатиться, и при несовместимых изменениях
(переименование колонки, смена типа, удаление `NOT NULL`) может либо потерять
данные, либо тихо отказаться выполнять команду. Для базы с реальными клиентами
это риск. Миграции хранят каждое изменение схемы как файл `.sql`, который
можно прочитать, проверить и откатить.

## Обычный рабочий процесс (с этого момента)

1. Поменял что-то в `db/schema.ts`.
2. `npm run db:generate` — drizzle сравнит схему с последней миграцией и
   создаст новый файл `db/migrations/000X_<имя>.sql` с нужными `ALTER TABLE`.
3. Открой этот файл и прочитай его — drizzle обычно угадывает правильно, но
   при переименовании колонки иногда предлагает "удалить + создать новую"
   вместо `RENAME COLUMN` — это удалит данные в проде. Если видишь `DROP
   COLUMN` там, где должен быть `RENAME` — почини SQL руками перед применением.
4. `npm run db:migrate` — применяет миграцию к реальной базе (адрес из
   `DATABASE_URL` в `.env`).
5. Закоммить и файл миграции, и обновлённый `db/schema.ts` в одном PR.

`db:push:dev-only` — оставлен только для быстрых локальных экспериментов на
твоей машине, когда не жалко потерять локальные данные. **Не используй его на
Railway или любой базе с реальными данными.**

## Важно: как подключить миграции к уже существующей Railway-базе

В этом репозитории создана **baseline-миграция**
(`db/migrations/0000_baseline.sql`) — она описывает текущую схему как набор
`CREATE TABLE`. На твоей реальной Railway-базе эти таблицы **уже существуют**
(их создал `db push` раньше). Если просто запустить `npm run db:migrate` —
он попытается выполнить `CREATE TABLE`, получит ошибку "table already exists"
и упадёт.

Это нормально и ожидаемо — нужен один разовый шаг, чтобы сказать drizzle
"эта миграция уже как бы применена, считай её сделанной, без реального
выполнения SQL":

```bash
npx drizzle-kit migrate --skip
```

Если такого флага в твоей версии `drizzle-kit` нет (версии иногда отличаются),
альтернативный способ — вручную создать в БД таблицу `__drizzle_migrations`
(drizzle создаёт её сам при первом обычном запуске `db:migrate` на пустой
базе) и вставить туда запись о выполнении `0000_baseline`, с тем же hash, что
указан в `db/migrations/meta/_journal.json`. Если не уверен в этом шаге —
сначала попробуй на **копии** базы (Railway позволяет сделать snapshot/fork),
не на проде напрямую.

После этого разового шага — `0000_baseline` помечена как выполненная, и
каждая следующая миграция (`0001_...`, `0002_...` и далее) применяется к
реальной базе как обычно через `npm run db:migrate`.

## Бэкап перед каждой миграцией в проде

Railway даёт снапшоты MySQL — перед `npm run db:migrate` на проде сделай
снапшот вручную (или включи автоматические, если есть такая опция в текущем
плане Railway). Это не отменяет необходимость читать сгенерированный SQL
перед применением (шаг 3 выше) — снапшот это просто страховка на случай,
если что-то пошло не так несмотря на проверку.
